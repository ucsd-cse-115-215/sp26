From LLM to Agent

NOTE(joe): Maybe this is lecture content. Thinking about the overall flow of the course I feel like this is a good story

The fundamental component of a GenAI system based on large language models is “tokens in, tokens out” – the system produces a stream of resulting text or data based on completions of the input.

AI agent systems do a lot of things, but one fundamental feature is mixing token output with structured tool use output. That is, rather than simply interpreting prompts and producing text or data for programs to consume, they have built-in functionality for taking actions.

This is not magic: with what we've learned we can imagine building an agent out of the simple kinds of prompts we've built before. A simple version is that we could ask the agent to generate a bash script, and then run that bash script programmatically (with exec or similar). The key feature is that the LLM generates responses that are interpreted as code or actions, and the agent system is responsible for taking those actions. That means the agent system needs to have a well-specified set of possible actions, often called “tools”, and the LLM needs to be prompted to generate valid tool instructions.

Without the surrounding agent system, the LLM can take no action. The agent system is responsible for enabling the actual actions taken, and the underlying LLM is still “just” generating a bunch of tokens. With an agent, we have chosen to write a program that takes those results and turns them into actions that affect the world, just like we do any time we make a system call from a traditional program.

Programming an Agent

We could try to build an agent from scratch, but agents are well-understood enough that there are well-specified APIs for creating them.

For example, this small project uses the OpenAI agents API along with a Language Server Protocol implementation for Python as a tool to create an agent that can do small refactorings on Python programs.

The key thing about this example agent is that it provides a chat-based interface to a well-understood problem (refactorings). The actions are expressed in terms of the LSP, which does structured edits rather than freeform textual edits from an LLM.

Your task for this assignment is to build an agent that:

  • uses tools
  • is focused on one small set of tasks
  • has some constraints and required user confirmations/interactions

There are a few options here, we offer these as starting points. As usual you are free to pick any project that satisfies the requirements below.

  • You could add an agent-based interface to one of your past two projects, turning the core functionality into an agent-based flow
  • An agent that helps grade an assignment from an intro programming course, prompted with the assignment description and using tools to work with the student code and give a grade according to a rubric.
  • An agent that helps a student un-f&$#k a git repository that is in a strange state, equipped with filesystem and git commands to save work and put things back in a reasonable state
  • A tutor for a narrow topic in a course, for example learning UNIX terminal commands, equipped with tools for inspecting the filesystem and checking that work is done correctly
  • An agent that can pan and zoom around a map equipped with API like the Google Maps API in response to text prompts to search for locations, look for information on the map, and so on

Requirements

  • Your project should use an agent API. You can make requests to it directly or use a SDK like the one linked above. Don't build an agent from scratch.
  • In terms of operations, you should have:
    • An operation that your agent can take safely on its own with no user confirmation – reading data, making a query, etc.
    • An operation that requires user confirmation – actually changing a file, giving away an answer as a tutor, or giving a response that would be hard to “take back”
    • An operation or kind of data that is disallowed by the agent. For example, asking the agent for answers directly in the tutoring system, or doing destructive actions in git that would lose work, and so on.
  • You should also include a script or other code that tests your agent programmatically. Include a programmatic test for at least one of each of the scenarios above.
  • Include in the repository README the transcript or description of how to reproduce an interaction where your agent performs very well, to showcase everything it can do
