Course Components

There are three main parts of the course that have a grade: Assignments, Peer Review and Feedback, and Oral Exams that are related.

Assignments and Reviews

The course has several assignments that involve programming and writing.

Each assignment will have two deadlines – an initial deadline, and a post-review deadline. Specific assignments will have different rules for working individually or in groups.

At the initial deadline, your goal is to produce a working implementation of the project description that you can demo, and others can try and give feedback on. The course staff will give some limited feedback during this period, and assign a tentative score from 0-4.

Your work will then be shared, presented, or demoed (depending on the specific week), and other groups will give feedback. This may include:

  • Code feedback
  • Bug reports
  • Feature requests

Specific assignments will have precise rubrics for what to provide. The course staff may also provide feedback in this form.

Then, for the post-review deadline, you should respond to the feedback. Again, we will develop specific requirements for each one of the assignments, but required responses may include fixing some or all of the repored bugs, adding one or more of the requested features, and describing or updating code that people were confused about.

The course staff will combine the initial submission score and your response to the feedback into an overall 0-4 score for the assignment.

Reviews you Write

As described above, you will conduct reviews of other groups' work on a regular basis. These also form a part of your grade, and are expected to be thoughtful, thorough, and high-quality. We want them to represent your genuine human feedback on the programs you work with.

Some reviewing will happen in-person, and other reviewing will be asynchronous. For reviews you write on your own, we ask that you not use generative AI to generate you r reviews. To motivate this, we will deduct substantial credit (give 0s) if we find obvious factual errors in reviews (AI hallucinated or otherwise).

Specific assignments will have specific instructions on review prompts, reflections, and feedback to give.

Reviewees (the students whose work is reviewed) will give a score for the usefulness and thoughtfulness of the feedback. The course staff will take this feedback-on-feedback into account when assigning a 0-4 reviewing score on each reviewing pass.

Oral Exams/Demos

Twice during the quarter you will meet with the course staff for an oral exam/demo where you demonstrate your assignment, show the feedback you got, and talk through how you responded to it. The first will be in week 5 or 6 and the second in week 10. In finals week, you'll have the chance to make up an oral exam/improve your score.

Your grade on the oral exams will be from 0-4; if you score less than 4 on the oral exam we'll give you feedback about what you could improve in a retry.

Grading

Each component of the course has a minimum achievement level to get an A, B, or C in the course. You must reach that achievement level in all of the categories to get an A, B, or C.

  • A achievement:
    • ≥85% of assignment points (e.g. likely 4 assignments at 4 points each, so 14/16 points)
    • ≥85% of review and feedback points (e.g. likely 4 assignments at 4 points each, so 14/16 points)
    • ≥6/8 oral exam/demo points
  • B achievement:
    • ≥75% of assignment points
    • ≥75% of review and feedback points
    • ≥5/8 oral exam/demo points
  • C achievement:
    • ≥60% of assignment points
    • ≥60% of review and feedback points
    • ≥4/8 oral exam/demo points

Below C achievement (in any category) is an F/No Pass.

Requests to change this grading policy (for a specific student or class-wide) will be denied with a link to this syllabus section. Consider this: we may, as instructors, decide for academic reasons that the most accurate way of assigning letter grades in the class needs to change (and we tend to only make changes that improve letter grades relative to this starting policy). However, it would be inappropriate for us to do so in response to student requests: that could create an appearance that we give students the grades they ask for rather than the grades they earned.

Social Media Monitor Assignment

Assignment goals:

  • Use basic LLM APIs for text processing
  • Learn to balance cost and quality by combining LLMs with cheaper processing stages
  • Build an eval harness for an LLM-powered application

Large-Scale Semantic Text Processing

One of the most compelling capabilities of LLMs is their ability to understand text at a deep semantic level. This enables a new kind of automation: pipelines that perform non-trivial classification and filtering over large volumes of text.

For example, this repository shows a short example of monitoring public posts on BlueSky to identify and collect original poetry.

While hosted LLM APIs (such as the OpenAI API) make it easy to get a prototype of such a pipeline running, there are two fundamental challenges:

  • Cost. LLMs are expensive — both in dollar terms for hosted APIs and in compute for self-hosted models — and more capable models cost more. To process data at scale, you need a multi-stage pipeline that progressively funnels the data through increasingly capable (and expensive) filters.
  • Reliability. LLMs are imperfect and unpredictable, so you need an eval harness to measure how well your pipeline performs and to systematically improve it.

A Reliable Social Media Monitor

Your task for this assignment is to build a social media monitoring system. You choose which platform or content source to monitor and what kind of content to look for. Our BlueSky poetry detector is just an example; other ideas include:

  • As a source, you could try Mastodon, HackerNews, Reddit, Telegram, or even GitHub. BlueSky is particularly convenient because it has a public firehose API that does not require an account or API key.
  • You might be interested in monitoring for:
    • internship postings tailored to your interests
    • academic paper announcements in a niche area
    • event announcements in your city

You may have your own ideas — please be creative!

Monitoring can be either online (content is streamed in and relevant items are displayed as they appear) or offline (content is downloaded at regular intervals and filtered in batch). There are no requirements on the UI: it can be a CLI application (like our poetry detector) or it can collect results into a table or dashboard.

No matter what you pick, there are some requirements for what you build:

Multi-Stage Pipeline

  • Whether online or offline, your monitor should be designed to scale to large amounts of content.
  • As a result, it won't be economical to send everything to a capable LLM; you will need to build a multi-stage pipeline.
  • You can be creative with what stages you use — this will depend on your domain. Common choices include (in order of increasing cost and precision):
    • Symbolic filtering by metadata (channel, language, location, etc.)
    • Keyword filtering
    • Embedding similarity (how close is this content to a set of known positive examples); computing embeddings requires an API call, but embeddings are orders of magnitude cheaper than LLM inference
    • A cheap/small LLM targeting high recall (and low precision), e.g. by tuning a classification threshold

Evaluation and Tuning

  • You need to build an eval harness that lets you measure your pipeline's performance and compare different configurations.
  • The eval harness should measure both quality (precision/recall) and cost.
  • You will need a gold dataset of at least 50 labeled examples to measure quality. Because your task and data source are unique to your project, you will need to build this dataset yourself.
  • Invest in tooling that makes labeling data and comparing pipeline configurations easy.
  • Once you have the eval harness and gold dataset, compare different model parameters and prompting techniques (personas, few-shot examples, explanations/CoT). Think about which metric to optimize for your domain: F1 at a given max cost? Recall at a given minimum precision and max cost? Cost at a given minimum recall? Different metrics make sense for different applications.

Stretch Goal

As a stretch goal, try using prompt tuning frameworks (DSPy) or "semantic operator" frameworks (Lotus, Palimpzest) to implement your pipeline and compare its performance with your manually tuned version.

Document Scanner Assignment

Assignment goals:

  • Create and manage prompts with multi-modal queries
  • Create a useful standalone application around core generative AI functionality

More than Text

Modern Generative AI systems are capable of working with much more than text. They support multi-media queries and are able to “read” and “see” documents like PDFs and images, and extract structured information from them.

For example, this repository shows a short example of taking an image of a receipt as input and prompting OpenAI to turn it into a CSV file.

This capability comes with some interesting consequences:

  • Like all other interactions with generative AI, the results are nondeterministic. Run the CLI example above several times -- you will see differences in the reported information from the receipt on each run. In addition,
  • Sourcing authentic test data can be more work. It's not obvious how to create many examples of images or PDFs for the program, while deploying an application that uses this functionality may end up seeing a wide variety of inputs from users.
  • Applications that work with user images and documents should have extra concerns about privacy and security: documents can unintentionally have extra information that users may not think about or intend to be shared or stored (account numbers, addresses, names, and so on).

An Application, not a Demo

Your task for this assignment is to pick a domain involving documents (receipts, transcripts, handwritten notes, chalkboard scans, screenshots – many options!) and write a useful application around it.

Here are just a few ideas:

  • A receipt tracking app that allows a user to upload receipt files (or directly take photos with a camera) and show them there spending on different items over time.

  • A note-tracking app that allows a user to take screenshots of the chalkboard or whiteboard after a class and get text-based notes and organize them throughout the quarter

  • A flashcard app that allows a user to upload a photo or scan of class handouts or lecture notes and automatically generates study flashcards from the key terms and concepts.

  • An app for tutors and TAs that allows them to take a photo of student work (e.g. their screen, which may have many windows open including some code and error messages) and helps them copy or run the code in another environment, as well as keep a journal of what happened with different students during their office hours, author frequent issues to send to their team, etc.

You may have your own, please be creative!

No matter what you pick there are some requirements for what you build:

User-facing Features

  • The application must have a GUI (we recommend web-based). It's fine if it's a demo that runs on localhost on a laptop, but the course staff should be able to easily clone and install it on a reasonable system themselves.
  • Support users uploading documents. We should be able to easily try our own document on your system.
  • Prompt GenAI to get structured information out of the document (sending the document to some backend)
  • Involve the user in verifying and clarifying ambiguity in the structured information, allowing the user to edit or correct it
  • Store the structured information somewhere (NOT the document itself)
  • Do some useful tasks for the user that involve one or more of:
    • Presenting the information in a new or different way than the original document
    • Aggregating, summarizing, or visualizing data from across all the documents that have been uploaded
    • Using some of the structured information to drive another GenAI query that does some useful work for the user

Engineering and Testing

  • You should have at least 20 examples of documents for your application. You can make them or find them yourself, and they can by synthetic.
  • You should have at least 5 examples of “near miss” documents for your application, where things don't work, and aren't really intended to work, but a user might plausibly try it. This could be a really bad scan, a garbled image, a handwritten receipt, a screenshot of Facebook rather than student work, etc.
  • The Gen AI prompting part of the system should be separately testable – that is, you should be able to run each of your example documents through the part of your system that generates structured answers without starting up the whole program. You should have an automated system that checks that the 20 “good” examples produce “good enough” results when run through the system, and the 5 “near miss” examples do not.

Agents and Tools

Assignment goals:

From LLM to Agent

NOTE(joe): Maybe this is lecture content. Thinking about the overall flow of the course I feel like this is a good story

The fundamental component of a GenAI system based on large language models is “tokens in, tokens out” – the system produces a stream of resulting text or data based on completions of the input.

AI agent systems do a lot of things, but one fundamental feature is mixing token output with structured tool use output. That is, rather than simply interpreting prompts and producing text or data for programs to consume, they have built-in functionality for taking actions.

This is not magic: with what we've learned we can imagine building an agent out of the simple kinds of prompts we've built before. A simple version is that we could ask the agent to generate a bash script, and then run that bash script programmatically (with exec or similar). The key feature is that the LLM generates responses that are interpreted as code or actions, and the agent system is responsible for taking those actions. That means the agent system needs to have a well-specified set of possible actions, often called “tools”, and the LLM needs to be prompted to generate valid tool instructions.

Without the surrounding agent system, the LLM can take no action. The agent system is responsible for enabling the actual actions taken, and the underlying LLM is still “just” generating a bunch of tokens. With an agent, we have chosen to write a program that takes those results and turns them into actions that affect the world, just like we do any time we make a system call from a traditional program.

Programming an Agent

We could try to build an agent from scratch, but agents are well-understood enough that there are well-specified APIs for creating them.

For example, this small project uses the OpenAI agents API along with a Language Server Protocol implementation for Python as a tool to create an agent that can do small refactorings on Python programs.

The key thing about this example agent is that it provides a chat-based interface to a well-understood problem (refactorings). The actions are expressed in terms of the LSP, which does structured edits rather than freeform textual edits from an LLM.

Your task for this assignment is to build an agent that:

  • uses tools
  • is focused on one small set of tasks
  • has some constraints and required user confirmations/interactions

There are a few options here, we offer these as starting points. As usual you are free to pick any project that satisfies the requirements below.

  • You could add an agent-based interface to one of your past two projects, turning the core functionality into an agent-based flow
  • An agent that helps grade an assignment from an intro programming course, prompted with the assignment description and using tools to work with the student code and give a grade according to a rubric.
  • An agent that helps a student un-f&$#k a git repository that is in a strange state, equipped with filesystem and git commands to save work and put things back in a reasonable state
  • A tutor for a narrow topic in a course, for example learning UNIX terminal commands, equipped with tools for inspecting the filesystem and checking that work is done correctly
  • An agent that can pan and zoom around a map equipped with API like the Google Maps API in response to text prompts to search for locations, look for information on the map, and so on

Requirements

  • Your project should use an agent API. You can make requests to it directly or use a SDK like the one linked above. Don't build an agent from scratch.
  • In terms of operations, you should have:
    • An operation that your agent can take safely on its own with no user confirmation – reading data, making a query, etc.
    • An operation that requires user confirmation – actually changing a file, giving away an answer as a tutor, or giving a response that would be hard to “take back”
    • An operation or kind of data that is disallowed by the agent. For example, asking the agent for answers directly in the tutoring system, or doing destructive actions in git that would lose work, and so on.
  • You should also include a script or other code that tests your agent programmatically. Include a programmatic test for at least one of each of the scenarios above.
  • Include in the repository README the transcript or description of how to reproduce an interaction where your agent performs very well, to showcase everything it can do
